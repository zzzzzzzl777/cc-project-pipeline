# Stage Prompts

Use these prompts when the user wants to hand work from one CC session to another.

## Intake CC

```text
请使用 cc-project-pipeline 做项目 intake。只读，不要删除或修改任何现有文件、文档、数据库、看板，也不要给任何人发消息。

任务背景：
<粘贴原始需求、项目目录、相关链接、截图说明>

要求：
1. 先收集证据，再总结需求。
2. 根据任务涉及的领域，使用对应的 domain skill。
2a. 如现有工具明显不够，可搜索本地 skill、官方文档、GitHub 开源项目，评估是否有更合适的方法或工具（仅评估，不安装——安装在执行阶段进行）。
2b. 默认先评估再采用；只有当工具能明显提升质量/效率/验证强度，且可隔离、低风险使用时，才允许安装。
2c. 不允许系统级安装、修改现有项目依赖/锁文件/环境配置、运行来源不明脚本。
3. 区分：数据/材料中直接存在的信息、被请求但缺失的信息、可通过间接证据推断的信息。
4. 列出关键待决策项：业务目标、核心定义/口径、范围、过滤条件、交付格式、优先级；先从已有材料和领域惯例中自查解决。
5. 只有自查后仍无法可靠解决，且会影响结果/执行/交付安全的问题，才列为 blocking questions；最多 3 个，每个说明为什么重要。
6. 对能通过逻辑性或可解释性策略解决的问题，按顺序尝试：查定义、推导、历史对齐、一致性检查、敏感性分析、多方案并列。
7. 多个标准都合理且无证据偏向时，并列展示各标准下结论会如何变化，不要默认选一个。
8. 受众建模：决策者是谁、要做什么决策、技术理解深度、已有认知（如可获知）。
9. 假设注册表：列出初始假设，标注类型（definitional/data/domain/technical）和验证状态。
10. 输出 01_intake.md：背景、目标、受众模型、已知输入、核心定义、信息分类（直接/缺失/可推断）、假设注册表、source coverage log（查了什么、用什么关键词/路径/时间范围、找到了什么、哪些来源不可达、哪些还没查）、已自解决策、待确认问题、交付物、风险。
11. 不要执行正式分析或产出最终报告。
12. 红线：不修改/删除现有文件；不写数据库/看板/生产系统；不发送任何消息。
```

## Planner CC

```text
请使用 cc-project-pipeline 只做计划，不要执行。

输入材料：
- 原始需求：<粘贴或给路径>
- Intake brief：<01_intake.md 路径>
- 项目目录：<路径>

要求：
1. 对照原始需求和 intake，判断信息是否足够。
2. 输出 02_plan.md，计划必须让另一个 CC 可以直接执行。
3. 写清楚范围、非目标、要用的 skills、输入来源、source coverage summary（intake 已查范围、仍需补查范围）、执行步骤、验证步骤、交付物、停止条件。
3a. 如预计需要额外工具/skill，写明：为什么现有方法不够、候选工具是什么、只搜索还是需要安装、安装是否可隔离、风险边界。
4. 每个步骤标注预估时间和 session 开销，给出总项目规模预判。
5. 每个关键决策写：决策、依据、解决策略、替代方案、选择理由、剩余不确定性、信心等级。
6. 先自查；自查后仍无法可靠解决且会影响结论时，最多列 3 个 blocking questions。
7. 多个合理标准/口径时，并列设计多标准/敏感性结果，不合并成单一方案。
8. 如果推荐涉及高风险决策（花钱、产品变更、不可逆操作等），计划必须包含敏感性分析步骤。
9. 写明数据验证门禁检查清单：覆盖范围、结构、量级、成熟度/完整性、一致性、已知坑点。
10. 列出需要嵌入执行阶段的自动化断言。
11. 假设注册表（从 intake 继承并补充），标注验证计划。
12. 如果 V2+ 迭代，写版本变更摘要。
13. 红线：不修改/删除现有文件；不写数据库/看板/生产系统；不发送任何消息。
14. 外部系统默认只读；如需写入或发消息，标记为阻塞。
```

## Data Validator CC

```text
请使用 cc-project-pipeline 只做数据验证门禁，不要执行分析。

输入材料：
- Plan：<02_plan.md 路径>
- 项目目录：<路径>
- 数据源说明：<路径或描述>

要求：
1. 按 plan 中的数据验证检查清单逐项检查。
2. 每项输出 pass/fail + 证据（查询结果、行数、字段列表等）。
3. 只做只读检查，不修改任何数据或文件。
4. 输出 02b_data_validation_report.md。
5. 门禁判断：proceed / loop back to plan / proceed with caveats。
6. 如果 loop back，列出需要在 plan 中调整的具体项。
```

## Executor CC

```text
请使用 cc-project-pipeline 执行这个项目。

输入材料：
- 原始需求：<粘贴或给路径>
- Intake brief：<01_intake.md 路径>
- Plan：<02_plan.md 路径>
- 项目目录：<路径>

要求：
1. 先审查计划，不要盲从；如果计划和证据冲突，先修正计划并说明。
2. 执行前必须先通过数据验证门禁。关键检查不通过时，回到计划阶段调整。
3. 按阶段执行，保留 03_execution_log.md。
4. 根据任务领域调用对应 domain skill。
4a. 如本地工具不足，可搜索并评估额外 skills / GitHub 项目 / 官方库；优先复用成熟方案，不重复造轮子。
5. 所有结论要有文件、数据、命令或链接证据。
6. 保留 decision log、assumption registry 和 source coverage log，执行中持续更新。
7. 嵌入自动化断言，断言失败时停下排查，不要静默跳过。断言阈值必须可解释——有数据支撑或逻辑推导均可，不允许使用无解释的任意数值。
8. 先自查后问用户；能通过逻辑性/可解释性策略解决的不确定性，先用策略解决。
9. 如果推荐涉及高风险决策，必须做敏感性分析。
10. 执行中发现证据与计划假设冲突时，触发 re-plan：停止受影响的分支、记录冲突、修订计划版本、提交用户批准后再继续。
11. 涉及推断的结论，标注类型（direct/derived/proxy/unsupported）和限制。
12. 红线：不修改/删除现有文件；不写数据库/看板/生产系统；不发送任何消息。
13. 需要修正版产物时，创建新文件，不覆盖原文件。
14. 显而易见的结论直接陈述，不做多余分析；把精力集中在真正影响结果的地方。
15. 如果任务可能超时，分段完成并写好交接，不要为了赶时间降低质量或跳过步骤。
16. 如果使用或安装了额外工具/skill，记录来源、版本、安装方式、是否隔离、用途、为什么值得采用，以及失败时的 fallback/cleanup。
17. 最后列出新产物路径和验证结果。
```

## Reviewer CC

```text
请使用 cc-project-pipeline 做严格 review。不要直接修复，除非我另说。

输入材料：
- 原始需求：<粘贴或给路径>
- Intake brief：<01_intake.md 路径>
- Plan：<02_plan.md 路径>
- 数据验证报告：<02b_data_validation_report.md 路径>
- Execution log：<03_execution_log.md 路径>
- 产物路径/链接：<列表>

要求：
1. 不要只看最终产物；必须对照原始需求、intake、plan、执行证据和产物。
2. findings 放最前面，按 P0/P1/P2 排序。
3. 必须逐项完成 5 维度强制审查，每个维度标注 PASS / PASS_WITH_CAVEATS / FAIL：
   a. 逻辑一致性：计数错误、构造性循环、因果语言、方向一致性。
   b. 数据可解释性：均值比率陷阱、聚合层级混淆、偏差格式、变换披露。
   c. 决策点逻辑：证据链完整性、过度简化、阈值可解释性。
   d. 数据准确性：叙述-表格交叉验证、算术正确、求和完整、跨表一致。
   e. 口径准确性：定义稳定性、来源对齐、跨段一致、版本漂移。
4. 检查决策严谨性：关键决策是否有证据、解决策略、替代方案；是否先自查后问用户。
5. 检查推断安全：缺失信息有没有被当成事实；是否标注为间接推断/待验证。
6. 检查红线合规。
7. 检查数据验证门禁是否通过；跳过的标记为 P0。
8. 检查假设注册表：高影响 untested 假设用作核心结论依据 → P1。
9. 检查自动化断言：失败被静默跳过 → P0。
10. 检查敏感性分析：高风险推荐未做 → P1。
11. 检查可视化准确性：面向外部受众的误导性图表 → P1。
12. 交付判断：READY_TO_DELIVER / READY_WITH_CAVEATS / NOT_READY / INSUFFICIENT_DATA。任何维度 FAIL → NOT_READY。
13. 检查 source coverage 是否足以支撑结论：哪些关键来源已覆盖、哪些未覆盖、哪些不可达、这会如何影响置信度。
14. 检查额外引入的工具/skill 是否真的提升了结果质量或效率，是否带来了隐性风险或不可复现性。
15. 如果没有问题，也要说明未验证的风险。
16. 输出 04_review.md。
```

## Fixer CC

```text
请使用 cc-project-pipeline 修复 review 问题。

输入材料：
- Review：<04_review.md 路径>
- 原始需求、intake、plan、execution log：<路径>
- 项目目录：<路径>

要求：
1. 只修 review 中的问题，不扩大范围。
2. 不要修改原文件；创建新的修正版文件。
3. 不写数据库/看板/生产系统，不发送消息。
4. 每修一个问题写入 05_fix_log.md：问题、改法、验证结果、剩余风险。
5. 修完后重新跑受影响的自动化断言，确认通过。
6. 检查修复是否影响假设注册表一致性，如有变化则更新。
7. 如果修复改变了核心方法论，视为 re-plan trigger 而不是简单修复。
```

## Final Check CC

```text
请使用 cc-project-pipeline 做最终交付检查。

输入材料：
- 原始需求：<粘贴或给路径>
- Intake/plan/data validation/execution/review/fix log：<路径>
- 最终产物：<路径或链接>

要求：
1. 检查每个承诺产物是否存在、非空、可打开。
2. 检查关键结论是否能追溯到证据。
3. 检查外部链接是否可访问。
4. 检查红线合规。
5. 检查推断安全：直接事实、间接推断、unsupported claims 是否分开。
6. 检查决策严谨性和问题处理。
7. 检查假设注册表最终状态：高影响 untested 假设是否标注为 caveat。
8. 检查自动化断言通过率。
9. 检查敏感性分析完整性。
10. 检查可视化准确性。
11. 如果 V2+ 迭代，检查版本变更摘要完整性。
12. pre-delivery gate：逐项确认需求匹配、逻辑、决策严谨性、数据质量、推断安全、可解释性、可执行性、可复现性、交付安全。
13. 交付判断：READY_TO_DELIVER / READY_WITH_CAVEATS / NOT_READY / INSUFFICIENT_DATA。
14. `READY_TO_DELIVER` 只是内部质量状态，不是外部交付许可。到达任何 verdict 后必须停止，把 verdict、产物和剩余 caveats 呈现给用户，等待用户明确批准后，才能执行任何外部交付动作。
15. 如过程引入了额外工具/skill，检查其依赖是否可复现、来源是否可信、是否改变了结果解释边界。
16. 输出 06_final_check.md 和 07_pre_delivery_gate.md。
17. 如果不是 READY_TO_DELIVER，列出必须修复的问题。
```
